## Deploying AKS on Azure GCC High for DoD IL4 Workloads: A Field-Tested Implementation Guide

If you're working in a DoD or federal civilian environment and you've been tasked with running containerized workloads at Impact Level 4 on Azure Government, this guide is for you. I've architected and deployed AKS clusters across multiple GCC High tenants for IL4 and IL5 programs, and I want to give you the exact playbook I use — not a sanitized overview, but real commands, real control mappings, and real pitfalls I've personally walked into.

Let's get into it.

---

## Why AKS on GCC High for IL4 Matters

Azure Government Community Cloud High (GCC High) is Microsoft's sovereign cloud environment purpose-built for DoD contractors, defense agencies, and federal entities handling Controlled Unclassified Information (CUI) and data governed by ITAR, EAR, or DFARS 252.204-7012. IL4 authorization covers CUI and mission-critical DoD workloads that don't reach classified (IL5/IL6) thresholds.

Running AKS at IL4 means every layer of your stack — the API server, node pools, networking, secrets management, container registry, and runtime — must satisfy the DoD Cloud Computing Security Requirements Guide (SRG) and align with NIST 800-53 Rev 5 controls enforced through FedRAMP High and the DoD PA (Provisional Authorization).

The specific controls this architecture directly addresses include:

- **AC-2 / AC-3**: Identity and access enforcement via Azure AD (Entra ID) RBAC and Kubernetes RBAC
- **AU-2 / AU-12**: Audit logging via Diagnostic Settings → Log Analytics → Microsoft Sentinel
- **CM-6 / CM-7**: Configuration hardening via Azure Policy and OPA Gatekeeper
- **IA-2 / IA-5**: MFA and managed identity enforcement
- **SC-7 / SC-8**: Network boundary protection and encryption in transit
- **SI-2 / SI-3**: Vulnerability scanning via Microsoft Defender for Containers

## Architecture Overview Before You Write a Single Command

Before touching the CLI, lock in these architectural decisions for IL4 compliance:

1. **Private cluster only** — API server must not be publicly accessible
2. **Azure CNI** (not kubenet) — required for network policy enforcement at IL4
3. **Managed identities** — no service principals with client secrets
4. **Entra ID-integrated RBAC** — disable legacy local accounts
5. **Encryption at rest** — customer-managed keys (CMK) in Azure Key Vault Managed HSM
6. **Node OS hardening** — use the CIS-hardened node image or Azure Linux (Mariner)
7. **Private ACR** — container images pulled only from a GCC High-native private registry
8. **Defender for Containers** — enabled at the subscription level via Defender for Cloud

  --disable-public-fqdn

  --name $CLUSTER_NAME \\

Key compliance flags explained:
- `--os-sku AzureLinux`: Azure Linux (CBL-Mariner) has a significantly smaller attack surface than Ubuntu — critical for SI-2 (patch management) and CM-6
- `--disable-local-accounts`: Enforces Entra ID-only auth, satisfying IA-2 and AC-2
- `--enable-azure-rbac`: Maps Kubernetes RBAC to Azure RBAC roles, giving you a single IAM plane auditable in Azure Monitor
- `--enable-workload-identity + --enable-oidc-issuer`: Replaces pod-level service principal credentials with federated identity tokens — eliminates a massive secret sprawl problem I see in almost every IL4 audit
- `--enable-encryption-at-host`: Encrypts the OS disk temp disk and cache at the hypervisor level, satisfying SC-28

Azure Policy is your automated CM-6 and CM-7 enforcement layer. GCC High includes the built-in **Kubernetes cluster pod security restricted standards** initiative. I also layer in the **NIST SP 800-53 Rev 5** built-in policy initiative.

## Step 4: Secrets Management with Key Vault and Workload Identity

At IL4, secrets cannot live in Kubernetes Secrets in plaintext base64. You must integrate Azure Key Vault with the CSI Secrets Store driver, backed by a Key Vault instance using a customer-managed key and private endpoint.

```bash
# Create Key Vault with CMK and private access only
az keyvault create \
  --name kv-il4-aks-prod \
  --resource-group $RESOURCE_GROUP \
  --location $LOCATION \
  --sku premium \
  --enable-purge-protection true \
  --retention-days 90 \
  --default-action Deny \
  --bypass AzureServices \
  --enable-rbac-authorization true

# Disable public network access
az keyvault update \
  --name kv-il4-aks-prod \
  --resource-group $RESOURCE_GROUP \
  --public-network-access Disabled
```

For your pods to consume secrets, configure a `SecretProviderClass` that references the Key Vault using a workload identity federated credential — no client secrets in your YAML manifests.

```yaml
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: il4-kv-secrets
  namespace: app-workload
spec:
  provider: azure
  parameters:
    usePodIdentity: "false"
    clientID: "<USER_ASSIGNED_MANAGED_IDENTITY_CLIENT_ID>"
    keyvaultName: "kv-il4-aks-prod"
    cloudName: "AzureUSGovernmentCloud"
    objects: |
      array:
        - |
          objectName: db-connection-string
          objectType: secret
          objectVersion: ""
    tenantId: "<YOUR_GCCHIGH_TENANT_ID>"
```

## Common Pitfalls I've Seen in IL4 AKS Deployments

**1. Forgetting to disable public FQDN after enabling private cluster**
The `--enable-private-cluster` flag alone doesn't disable the public DNS record. You must explicitly call `az aks update --disable-public-fqdn` or set it in Bicep via `enablePrivateCluster: true` AND `privateDNSZone: 'system'` with `disableRunCommand: true`.

**2. Using kubenet instead of Azure CNI**
Kubenet does not support Azure Network Policy. Without network policy, you cannot enforce pod-to-pod segmentation — a hard requirement for IL4 multi-tenant workloads.

**3. Leaving local accounts enabled**
Many teams enable local accounts for "break-glass" access and then forget to rotate or restrict it. Under IL4, `--disable-local-accounts` should be non-negotiable. Use Entra ID Privileged Identity Management (PIM) for emergency access instead.

**4. Not scoping Defender for Containers to the subscription level**
Enabling Defender only at the cluster level misses registry scanning and runtime threat detection for dynamically created namespaces. Enable it at the subscription in Defender for Cloud.

**5. Missing the `cloudName` parameter in SecretProviderClass**
This one has burned multiple engineers I've mentored. If you deploy the CSI driver without specifying `cloudName: "AzureUSGovernmentCloud"`, your pods will attempt to reach the commercial Key Vault endpoint and fail silently or timeout. Always explicit in sovereign clouds.
